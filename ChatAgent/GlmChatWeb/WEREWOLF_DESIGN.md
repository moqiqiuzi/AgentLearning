# 狼人杀方案设计 v3

## 一、角色配置

### 1.1 人数支持：6-12 人

| 玩家数 | 狼人 | 预言家 | 女巫 | 猎人 | 守卫 | 村民 |
|---|---|---|---|---|---|---|
| 6 人 | 1 | 1 | 1 | 0 | 0 | 3 |
| 7 人 | 2 | 1 | 1 | 0 | 0 | 3 |
| 8 人 | 2 | 1 | 1 | 1 | 0 | 3 |
| **9 人（默认）** | **2** | **1** | **1** | **1** | **1** | **2** |
| 10 人 | 2 | 1 | 1 | 1 | 1 | 3 |
| 11 人 | 3 | 1 | 1 | 1 | 1 | 4 |
| 12 人 | 3 | 1 | 1 | 1 | 1 | 5 |

### 1.2 角色能力

| 角色 | 夜间能力 |
|---|---|
| 🐺 狼人 | 每晚商议杀一人 |
| 🔮 预言家 | 每晚查验一人身份（好人/狼人） |
| 🧪 女巫 | 一瓶解药（救人）+ 一瓶毒药（毒人），各用一次 |
| 🏹 猎人 | 被杀或被投票出局时可带走一人 |
| 🛡️ 守卫 | 每晚守护一人（不能连续守同一人） |
| 👤 村民 | 无特殊能力 |

## 二、游戏流程

```
选择人格 → 分配角色 → 【夜晚 → 白天】循环 → 胜负判定 → 可选保存回放
```

### 2.1 夜晚阶段（自动执行）

```
1. 守卫选择守护对象
2. 狼人商议击杀目标
3. 预言家查验一人
4. 女巫得知被杀者，决定是否用药
5. 天亮，结算结果
```

结算逻辑：
- 被杀者 == 被守护者 → 未死
- 被杀者被女巫救 → 未死
- 被杀者未被守未被救 → 死亡
- 女巫毒了某人 → 那人死亡
- 一夜可能死 0-2 人

### 2.2 白天阶段

```
1. 公布死亡（动画：天亮效果 + 死亡公告）
2. 遗言：死者发表遗言
3. 发言：存活玩家按座位顺序发言
4. 投票：每人投一票（平票则无人出局）
5. 被放逐者遗言
6. （猎人死亡）猎人开枪
7. 胜负判定
```

### 2.3 胜负条件

- **好人胜**：所有狼人被淘汰
- **狼人胜**：存活狼人数量 ≥ 存活好人数量

## 三、用户模式

### 3.1 三种身份

| 模式 | 说明 | 视角 |
|---|---|---|
| **上帝** | 纯观察，不参与游戏 | 看到所有信息（上帝面板 + 公开区） |
| **玩家** | 抽取角色，参与讨论和投票 | 只看自己该看的，夜间需手动操作 |
| **旁观者** | 纯看公开信息 | 只看白天发言和投票，不看作弊面板 |

### 3.2 玩家模式操作

- **夜间**：根据角色弹出操作面板（守卫选人守护、狼人选人杀、预言家选人查、女巫决定用药）
- **白天**：在聊天框发言参与讨论，投票时点击头像选择目标
- 其他所有操作与人格玩家一致

## 四、信息隔离

每个人格/用户的 API 调用只传入它应该知道的信息：

| 角色 | 知道 | 不知道 |
|---|---|---|
| 狼人 | 队友身份、杀谁 | 预言家/女巫/守卫的行动 |
| 预言家 | 查验结果 | 其他特殊角色身份和行动 |
| 女巫 | 被杀者、药水状态 | 守卫守谁、预言家查谁 |
| 守卫 | 自己守谁 | 狼人杀谁、女巫行动 |
| 猎人 | 无夜间信息 | 所有特殊角色 |
| 村民 | 无夜间信息 | 所有特殊角色 |

**所有人共享**：白天发言、投票结果、死亡公告。

### 5.3 动画效果

#### 夜晚/白天切换
```
夜晚：背景渐变为深蓝 → 星星粒子飘落 → 月亮升起
白天：背景渐变为暖黄 → 阳光扩散效果 → 云朵飘过
```

#### 死亡公告
```
玩家头像变灰 → 上浮消失 → 座位标记 ⚰️
（CSS: opacity 1→0 + transform translateY(-20px)）
```

#### 角色行动
```
- 守卫守护：蓝盾光圈动画环绕被守护者
- 狼人杀人：红色目光射线从狼人射向目标
- 预言家查验：紫色光球飞向目标，返回绿色（好人）/红色（狼人）
- 女巫用药：绿色/紫色药水瓶倒出动画
- 猎人开枪：橙色射击线 + 闪屏效果
```

#### 投票动画
```
每个投票显示为箭头：投票者 → 被投者
箭头依次出现（有延迟）
最高票者头顶出现 ❌ 标记
```

#### 开局翻牌
```
9 张卡牌背面朝上 → 依次翻转 → 显示人格 + 角色
（CSS 3D transform: rotateY）
```

## 六、技术实现

### 6.1 新增文件

```
GlmChatWeb/
  werewolf.py          游戏引擎（状态机、结算、胜负判定）
```

### 6.2 游戏状态机

```python
game = {
    "id": "game_xxx",
    "phase": "night" | "day",
    "sub_step": "guard" | "werewolf" | "seer" | "witch" | "announce" | "speech" | "vote" | "hunter",
    "round": 1,
    "config": {"total": 9, "user_mode": "god" | "player" | "observer"},
    "players": [
        {
            "name": "鲁迅",
            "role": "werewolf",
            "alive": True,
            "seat": 1,
            "is_user": False,
            # 角色特有状态
            "werewolf_partners": ["杠精"],
            "seer_checks": {},
            "witch_save_used": False,
            "witch_poison_used": False,
            "guard_last": None,
        },
        {
            "name": "你",
            "role": "seer",
            "alive": True,
            "seat": 5,
            "is_user": True,
            ...
        }
    ],

    },
    "public_log": [...],

```

### 6.3 路由

```
POST /api/werewolf/start
  Body: { "total": 9, "user_mode": "player", "personas": [...], "user_role": "random"|"seer"|... }
  快捷模式：personas 留空则随机选人

POST /api/werewolf/step
  推进一步 → SSE 流式返回（带事件类型）
  用户玩家模式时，需要先 POST 用户的操作

POST /api/werewolf/action
  用户玩家提交操作（守护谁、杀谁、查谁、用药、投票等）

GET /api/werewolf/status
  返回游戏状态（根据 user_mode 过滤信息）

POST /api/werewolf/reroll
  重新生成上一步

POST /api/werewolf/stop
  结束游戏

POST /api/werewolf/save
  保存完整回放记录到 chat.db 或 JSON 文件
```

### 6.4 SSE 事件类型

```
data: {"event": "phase_change", "phase": "night", "round": 2}
data: {"event": "action_start", "role": "guard", "player": "历史学家"}
data: {"event": "action_text", "player": "历史学家", "text": "我守护诗人..."}
data: {"event": "action_result", "role": "guard", "target": "诗人"}
data: {"event": "death", "players": ["苏轼"], "cause": "werewolf"}
data: {"event": "speech", "player": "鲁迅", "text": "我觉得..."}
data: {"event": "vote_cast", "from": "鲁迅", "to": "诗人", "reason": "..."}
data: {"event": "vote_result", "eliminated": "杠精", "votes": {...}}
data: {"event": "game_over", "winner": "village", "replay_id": "..."}
```

### 6.5 回放存储

游戏结束时用户点击「保存回放」，完整记录存入 SQLite：

```sql
CREATE TABLE IF NOT EXISTS werewolf_replays (
    id          TEXT PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id),
    config      TEXT NOT NULL,    -- JSON: 人数、角色配置、人格列表
    events      TEXT NOT NULL,    -- JSON: 所有 SSE 事件的完整记录
    winner      TEXT NOT NULL,    -- 'village' 或 'werewolf'
    created_at  TEXT NOT NULL
);
```

### 6.6 速度控制

| 档位 | 每步间隔 | 适用 |
|---|---|---|
| 慢 | 3 秒 | 认真看剧情 |
| 中 | 1.5 秒 | 默认 |
| 快 | 0.5 秒 | 快速推演 |

### 8.2 人设来源规则

```
开局选主题 → 取预写角色（>= 人数）→ 够用 → 直接用
                              → 不够 → AI 生成补位角色
```

- **预写角色**：固定在代码/配置文件中，prompt 经过打磨，质量稳定
- **AI 补位**：调 GLM API 根据主题风格 + 角色名生成 6 维人设（复用 `/api/personas/generate` 逻辑）
- **自由模式**：不选主题时，从用户已启用的人格中选（现有逻辑）

### 8.3 角色生命周期

- **游戏期间**：主题角色存在于内存，绑定到游戏 session
- **游戏结束时**：弹窗询问用户
